Features

Why Larapilot

AI agents are fast, but isolated prompts are not a product process. Larapilot turns your assistant into a disciplined squad.

A workflow, not prompt lore

Discovery → backlog → plan → implement → review → ship. Every phase has clear inputs, outputs, and guardrails enforced by the CLI.

Spec-driven

The spec → plan → implement loop repeats per increment with enforced status transitions.

Persistent artifacts

PRD, backlog, specs, plans, and mockups live in .larapilot/ — version-controlled.

Laravel-native

Artisan commands, Boost docs, schema, Tinker — not a separate toolchain.

Packaged design systems

Filament and Laravel Starter Kit references — tokens, components, and static HTML screens — copied to .larapilot/design-systems/ for consistent mockups.

Client & legacy intake

Structured folders for client materials, legacy codebases, and competitor research — honored by every skill from discovery onward.

MoSCoW on requirements

Every FR-XXX in the PRD carries Must / Should / Could / Won't — Mark tags them at inception; /larapilot-spec uses them to bootstrap or defer backlog items.

Agent-friendly CLI

Consistent CLI output your AI agent can rely on.

Architecture

How it works

Larapilot is a product process backed by files and a CLI — not a one-shot prompt. Three layers cooperate; a local dashboard and JSON API give humans a live read-only view of the same artifacts.

Layer What it is Role
Skills /larapilot-* in your AI editor (via Boost) Playbooks: personas, context to load, Artisan commands to run
Artisan CLI php artisan larapilot:* Validates input, persists artifacts, enforces workflow transitions
Artifacts .larapilot/ in your repo Source of truth: PRD, backlog, specs, plans, mockups, design systems, intake folders — version-controlled
Dashboard /larapilot in the browser (local only) Read-only board, PRD viewer, and spec/task detail — mirrors artifacts, no YAML editing
Workflow API /larapilot/api/* via HTTP client (local only) Read-only JSON for board, specs (filter by status), PRD — OpenAPI 3.1 + Swagger UI at /larapilot/api/docs

Your editor connects to two MCP servers: Laravel Boost (docs, schema, Tinker) and Larapilot (backlog and workflow). Skills orchestrate the conversation; the CLI guarantees consistency; artifacts survive between sessions. When you run the app locally, open the workflow dashboard at /larapilot to see backlog status at a glance without opening YAML files — or call the JSON API from scripts and integrations.

Workflow dashboard (local)

The dashboard is a dev-only companion to .larapilot/ — it reads the same files the CLI writes and never replaces them. Use it to orient yourself between skill runs, share progress with teammates, or sanity-check status before approving a review.

View URL What you see
Board /larapilot Kanban columns by status (TODO → DONE), completion metrics, links to each story
PRD /larapilot/prd Rendered PRD.md with table of contents — available after inception
Spec detail /larapilot/specs/US-XXX User story, acceptance criteria, and plan tasks for one backlog item

Available in local, staging, and testing; returns 404 in production. Refresh the page after each skill run to pick up status changes. Disable with LARAPILOT_DASHBOARD_ROUTE=false (this also disables the API — same access gate). For JSON endpoints and Swagger UI, see the Workflow API reference.

The process in two parts

Larapilot splits work into a project phase (once) and a spec loop (repeat per user story).

Phase Skills What you get
1. Shape the product /larapilot-inception → /larapilot-spec PRD + backlog with user stories (US-001, US-002 …)
2. Build each story /larapilot-plan → /larapilot-implement → /larapilot-review Laravel code + tests, one story at a time
3. Release (optional) /larapilot-ship Security gate + deploy + launch checks

Between plan and implement you can run /larapilot-design for UI mockups. After the backlog exists, use /larapilot-feature for single evolutive and /larapilot-bug for defect intake. When the backlog is stable, /larapilot-autopilot batch-plans and implements multiple specs. The CLI blocks invalid jumps — you cannot implement before planning or approve before review.

Discovery (inception)

Everything starts with /larapilot-inception. You bring a rough idea — one sentence is enough. Optionally place client documentation in .larapilot/client-materials/ and legacy project snapshots in .larapilot/legacy/ before starting — the team reads them first. When legacy content is present, Mark (with Sabrine) proactively asks whether to pursue a legacy rewrite/port. Mark (PM) asks for Project Kind (Personal, Website, Application), then the team runs a short guided interview — at most 3 questions per round, all skippable. Sebastian may ask for reference product URLs and run deepsearch (findings land in .larapilot/research/reference-products/). When rewriting legacy software, every feature and dataset stays in scope unless you explicitly defer it in the PRD. Mark assigns MoSCoW (Must, Should, Could, Won't) on each functional requirement in ## Functional Requirements — the per-FR scope lens that complements delivery target and backlog priority. When there is enough context, they write and validate .larapilot/docs/PRD.md. Inception does not create the backlog; that is /larapilot-spec.

Spec loop (per user story)

Each story in the backlog follows the same cycle:

  1. /larapilot-plan US-XXX — technical plan with tasks → PLANNED
  2. /larapilot-implement US-XXX — code and tests → REVIEW
  3. /larapilot-review US-XXX — you approve → DONE, or send back → TODO

See the full walkthrough for a concrete example from idea to shipped MVP. After the MVP is live, use feature and bug examples for day-to-day evolutive work.

Incremental work (post-MVP)

When the PRD and backlog already exist, you do not need full inception again:

  • /larapilot-feature "…" — one new capability with a short guided interview → new US-XXX spec
  • /larapilot-bug "…" — Sophia triages a defect → fix spec or rework on an existing story

Both commands reuse the normal spec loop (plan → implement → review). See practical examples with the exact prompts above.

Install Larapilot Quickstart

From your Laravel project root: composer require andreapollastri/larapilot --dev, then php artisan larapilot:install and php artisan boost:install. Enable Boost + Larapilot MCP servers in your editor. You are ready to invoke skills.

With php artisan serve running, /larapilot is already wired — it stays empty until inception and spec create the first artifacts.

No personas yet — install publishes guidelines (coding rules) and the ten /larapilot-* skill playbooks. Personas appear when you invoke a skill.

Examples

Feature & bug — practical walkthroughs

After inception and the first backlog, most day-to-day work is one evolutiva or one bug at a time. These skills run a short interactive interview (AskQuestion wizard, max 3 questions per round), then persist artifacts — same spec loop as greenfield work.

Which command when?

/larapilot-feature
New capability the product should gain — export, integration, UI improvement, admin screen. Requires .larapilot/docs/PRD.md. Mini-inception: MoSCoW, FR traceability, optional mockup-first.
/larapilot-bug
Something broken vs expected behaviour — regression, browser quirk, SSO failure. Sophia triages severity and environment, logs support/intake.md, routes to fix spec or rework.
/larapilot-spec (bulk)
Bootstrap the whole backlog from the PRD, or append several stories at once without the guided interview.
/larapilot-inception
Greenfield product or a vision pivot — full PRD discovery. Not for single evolutive or bug fixes.

Quickstart

Up and running in minutes

Install the package, enable MCP servers, and invoke skills in your AI editor.

Requirements

  • PHP ^8.3
  • Laravel ^12 or ^13
  • Laravel Boost ^2.0 (installed automatically with Larapilot)
  • An AI editor with MCP support (Cursor, etc.)

1. Install

Three commands, run from your Laravel project root. Laravel Boost is installed automatically as a Larapilot dependency — no separate composer require needed.

1a — Add the package:

Terminal
composer require andreapollastri/larapilot --dev

1b — Initialize Larapilot. Creates the .larapilot/ workspace in your repo:

Terminal
php artisan larapilot:install
Expected output
INFO  Larapilot installed successfully.

  - .larapilot/config.yaml
  - .larapilot/shared-runtime.md
  - .larapilot/client-materials/
  - .larapilot/legacy/
  - .larapilot/research/
  - .larapilot/design-systems/

Next: run php artisan boost:install (or boost:update --discover) to publish AI skills and guidelines.

1c — Publish skills via Laravel Boost. Makes the ten /larapilot-* skills (inception, design, spec, plan, implement, review, ship, autopilot) and the coding guidelines (not a skill) available to your AI editor:

Terminal
php artisan boost:install

Boost's installer asks which editors and agents you use, then publishes guidelines and skills for them. Already running Boost in the project? Use php artisan boost:update --discover instead to pick up the Larapilot skills.

On an already-installed project, larapilot:install fails fast and points you to larapilot:update (after upgrades) or --force (to overwrite config). larapilot:update never touches .larapilot/config.yaml.

2. Enable MCP servers

Your editor needs both servers: Boost for Laravel context (docs, schema, Tinker) and Larapilot for workflow state.

Server Command Args
laravel-boost php artisan boost:mcp
larapilot php artisan mcp:start larapilot

Concrete example for editors with a JSON MCP config (Cursor: .cursor/mcp.json, Claude Code: .mcp.json):

.mcp.json
{
  "mcpServers": {
    "laravel-boost": {
      "command": "php",
      "args": ["artisan", "boost:mcp"]
    },
    "larapilot": {
      "command": "php",
      "args": ["artisan", "mcp:start", "larapilot"]
    }
  }
}

3. Use skills in your AI agent

Skill Purpose
/larapilot-inception Product discovery → .larapilot/docs/PRD.md
/larapilot-feature One evolutiva → new US-XXX spec
/larapilot-bug Bug triage → fix spec or rework
/larapilot-design UI mockups → .larapilot/mockups/{spec}/
/larapilot-spec Backlog & user stories
/larapilot-plan US-001 Technical plan & tasks
/larapilot-implement US-001 Code, tests, review
/larapilot-review US-001 Human acceptance gate
/larapilot-ship OWASP gate + multi-platform deploy
/larapilot-autopilot Batch plan + implement

4. Run your first workflow

Once installed, open your AI editor and follow the end-to-end walkthrough — it shows every step from a one-line idea to shipped Laravel code, with concrete examples of PRD, backlog, plans, and code.

Short version: /larapilot-inception → /larapilot-spec → for each story /larapilot-plan → /larapilot-implement → /larapilot-review → optionally /larapilot-ship. After MVP: /larapilot-feature or /larapilot-bug — see examples.

5. Verify installation

Terminal
php artisan larapilot:doctor
Expected output
{
  "schema": "larapilot/v1",
  "kind": "doctor",
  "data": {
    "healthy": true,
    "checks": {
      "config": true,
      "shared_runtime": true,
      "backlog": false,
      "prd": false,
      "boost": true
    },
    "project_root": "/path/to/your-app"
  }
}

data.healthy is true when config, shared runtime, and Boost are all in place. On a fresh install backlog and prd are still false — that's expected: they turn true after /larapilot-inception (PRD) and /larapilot-spec (backlog). In local, browse the workflow dashboard at /larapilot or call the JSON API once artifacts exist.

6. Keep it updated

Boost publishes guidelines and skills as copies into your editor directories, so after upgrading the package one command brings everything current — the shared runtime doc, the Larapilot guidelines, and the /larapilot-* skills:

Terminal
composer update andreapollastri/larapilot
php artisan larapilot:update

larapilot:update never touches .larapilot/config.yaml — your project customizations survive — and re-runs boost:update with the editor and agent choices you already made during boost:install (no questions asked). Pass --skip-boost to refresh only the shared runtime and manage Boost publishing yourself.

To keep the project aligned automatically, hook it into Composer in your app's composer.json:

composer.json
"scripts": {
    "post-update-cmd": [
        "@php artisan larapilot:update --ansi"
    ]
}

From then on every composer update republishes runtime, guidelines, and skills — nothing goes stale silently. Run php artisan larapilot:doctor anytime to confirm the install is healthy.

Transition guards

The CLI blocks invalid transitions:

  • spec-start — requires PLANNED
  • spec-review — requires IN PROGRESS
  • spec-approve / spec-request-changes — require REVIEW
Skills

Eight skills, full coverage

Published via Laravel Boost and invoked in your AI agent. Boost also publishes guidelines (coding rules — not a skill). These ten /larapilot-* skills cover the workflow: inception, feature, bug, design, spec, plan, implement, review, ship, autopilot.

/larapilot-inception

Product Inception

Guided discovery interview — not a one-shot form. Mark opens with Project Kind (Personal, Website, Application); the team explores your idea conversationally, then writes and validates the PRD.

  • Branches: Personal (lean), Website (type + SEO/marketing), Application (full SaaS/platform discovery)
  • Intake: reads client-materials/ and legacy/ before the interview; Sebastian runs deepsearch on reference product URLs
  • Team: Mark, Jennifer, Benjamin, Sebastian, John, Aurora; Emma, Lauren, Violet join when relevant

/larapilot-feature

Feature / Evolutiva

Mini-inception for one new capability on an existing project — interactive AskQuestion rounds, then a single backlog spec. Requires a PRD; does not replace full inception.

  • Example: /larapilot-feature "Add PDF export for invoices"

/larapilot-spec

Backlog & Specs

Creates or extends the backlog with user stories from the PRD. Reads each FR's MoSCoW tag as the primary input for bootstrap vs deferral (falls back to delivery target for legacy PRDs).

  • Team: Tom (Requirements Analyst), Mark (PM); Sabrine for legacy parity

  • Priority: Must → HIGH (compliance → CRITICAL); Should → MEDIUM; Could → LOW
  • Output: backlog.yaml + specs/US-XXX.yaml with FR-XXX traceability
/larapilot-plan

Technical Planning

Detailed implementation plan with tasks, file changes, and test strategy → PLANNED.

  • Team: Tom, John, Sebastian, Aurora, Violet, Alex, Anne, Andrew; Emma & Lauren for public-facing specs; Sabrine for legacy porting

  • Vendor policy: Laravel first-party → Spatie → Filament & plugins (when chosen) → other vetted vendors — always maintained and secure

Implementation

Acceptance gate — Robert presents the increment and residual risks; judges delivered scope against FR MoSCoW tags and the PRD delivery target. You approve to DONE or request changes back to TODO.

  • Team: Robert (Code Reviewer); Sabrine checks legacy parity when applicable

Ship & Deploy

  • Team: Lars, Jack, Aurora, Violet, Emma & Lauren (public sites)

PRD functional requirements

Each requirement under ## Functional Requirements uses a level-3 heading and a MoSCoW line. Labels stay in English (Must, Should, Could, Won't) even when requirement text is localized. After inception, scope changes append rows to ## PRD Revision History — see PRD living document.

PRD.md (excerpt)
## Functional Requirements

### FR-001: User registration with email and password
**MoSCoW:** Must

### FR-002: SSO via Google Workspace
**MoSCoW:** Won't

Spec format

US-001.yaml body
#### US-001: User Registration

**Epic:** EP-001 | **Priority:** HIGH | **Points:** 5 | **Status:** TODO

**User Story**
As a new visitor,
I want to create an account,
So that I can access protected features.

**Acceptance Criteria**
- [ ] AC-001: Email validation enforced
- [ ] AC-002: Password minimum 8 characters

Language & validation

Artifacts can be in any language. The structure is fixed — only labels and content are translated.

Every section must use a markdown heading (## Title or **Title**). Plain prose that mentions “User Story” without a heading does not count.

Artifact Required sections Example (EN → IT)
PRD 6 level-2 headings ## Elevator Pitch → ## Sintesi
Spec body User Story, Demonstrates, Acceptance Criteria **User Story** → **Storia Utente**
Plan task Description ## Description → ## Descrizione

The validator first looks for known translations of these section names. If a heading is in another language, validation still passes when there are enough marked headings:

  • PRD — 6 headings with ## …
  • Spec body — 3 headings with ## … or **…**
  • Plan task — 1 heading with ## … per task

Workflow API

Read-only JSON endpoints under /larapilot/api/ — same access rules as the dashboard. Use from CI scripts, local tooling, or any HTTP client. Interactive docs at /larapilot/api/docs (also linked from the dashboard nav as API).

Endpoint Method Description
/larapilot/api/board GET Full Kanban snapshot: metrics, status order, columns with all user stories and task progress
/larapilot/api/specs GET List all backlog specs; optional ?status=TODO filter (case-insensitive)
/larapilot/api/specs/{code} GET Single user story with plan summary, tasks, workdir, and task progress
/larapilot/api/prd GET PRD markdown and parsed section headings (404 when no PRD exists)
/larapilot/api/openapi.json GET OpenAPI 3.1 machine-readable spec (server URL resolved from the current request)
